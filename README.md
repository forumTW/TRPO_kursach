## 1. ОБЩЕЕ ОПИСАНИЕ СИСТЕМЫ

- Полное название и назначение системы

Приложение представляет собой веб-систему для создания, прохождения и анализа тестов/квизов. Это образовательная платформа, где преподаватели могут создавать тесты и вопросы, а студенты — проходить тесты и получать результат и разбор ответов. Система реализована как монолитное Flask-приложение с серверной логикой, шаблонами и ORM-моделями для хранения данных.

- Какую реальную проблему решает

Система автоматизирует процесс проведения контрольных работ и тестирования знаний: хранит тесты, управляет доступом преподавателей и студентов, фиксирует попытки прохождения и автоматически подсчитывает баллы. Это устраняет ручную проверку ответов для простых тестов и даёт возможность получить статистику по результатам. Подходит для небольших учебных заведений или курсов, где требуется простой механизм онлайн-тестирования.

- Для кого предназначена (целевая аудитория)

Целевая аудитория — преподаватели и студенты учебных заведений (курсы, техникумы, отделения дополнительного образования). Преподаватели используют интерфейс для создания тестов и анализа результатов, а студенты — для регистрации, прохождения тестов и просмотра своих результатов. Административного интерфейса за пределами возможностей преподавателя в коде не предусмотрено.

- Основные возможности (кратко)

Создание предметов и тестов (quiz) преподавателями, добавление вопросов и вариантов ответов; прохождение тестов студентами с опциональным таймером; автоматический подсчёт результатов и хранение попыток; подробный разбор ответов для каждой попытки; страницы статистики по тестам для преподавателя.


## 2. ТЕХНОЛОГИЧЕСКИЙ СТЕК
Перечислено всё, что используется в проекте (на основе `requirements.txt` и кода).

Основной язык — Python. В `requirements.txt` указаны версии библиотек, совместимые с Python 3.x (очевидно ориентировано на современный Python 3.10+ / 3.11+, конкретная версия интерпретатора в проекте не фиксируется). Код использует синтаксис и библиотеки, совместимые с Python 3.

- Фреймворки и библиотеки (название + версия + для чего используется)

- `Flask==3.1.0` — основной веб-фреймворк для маршрутизации, обработки запросов и рендеринга шаблонов.
- `Flask-SQLAlchemy==3.1.1` — ORM-обёртка SQLAlchemy для работы с базой данных и декларативного описания моделей.
- `Flask-Login==0.6.3` — управление сессиями пользователей и декораторы `login_required`, удобная интеграция с моделью `User`.
- `Flask-WTF==1.2.2` и `WTForms==3.2.1` — указаны в зависимостях, но в коде формы реализованы вручную; библиотека подключена, но явных `FlaskForm` в `app.py` нет.
- `psycopg2-binary==2.9.10` — драйвер для подключения к PostgreSQL (используется, если в конфигурации задан `DATABASE_URL` для Postgres).
- `python-dotenv==1.0.1` — загрузка переменных окружения из `.env` (в коде вызывается `load_dotenv()`).
- `Werkzeug==3.1.3` — утилитарные функции и `generate_password_hash`/`check_password_hash` для хеширования паролей.

- СУБД и драйвер подключения

По умолчанию в конфигурации `app.config['SQLALCHEMY_DATABASE_URI']` указан шаблон для PostgreSQL: `postgresql://postgres:postgres@localhost:5432/quiz_db`. Для подключения к PostgreSQL в зависимостях присутствует `psycopg2-binary`. Однако в тестах используется SQLite in-memory (`sqlite:///:memory:`), поэтому проект поддерживает обе СУБД через SQLAlchemy.

- Фронтенд (HTML/CSS/JS фреймворки, библиотеки)

Фронтенд реализован серверными Jinja2-шаблонами (папка `templates`). Видна кастомная стилизация через CSS-классы (в `base.html` — базовая вёрстка и стили; файл не парсен подробно, но шаблоны показывают классы и иконки `bi` — Bootstrap Icons или схожие). JavaScript используется локально в шаблонах: обработчики выбора ответов и простой таймер на странице `take_quiz.html`. Отдельных фронтенд-фреймворков (React/Vue/Angular) не используется.

- Инструменты разработки и тестирования

`pytest==7.4.3` и `pytest-flask==1.3.0` используются для тестирования. В репозитории есть тесты в папке `tests/` с фикстурами в `conftest.py` и множеством функциональных тестов в `test_app.py`. Присутствуют CLI-команды Flask для инициализации БД и создания преподавателя (`flask init-db`, `flask create-teacher`).

- Конфигурация и переменные окружения

Переменные окружения загружаются через `python-dotenv` (`load_dotenv()`). В `app.py` используются `SECRET_KEY` и `DATABASE_URL` через `os.getenv`. Значения по умолчанию предоставлены (`dev-secret-key` и строка подключения к локальному Postgres). В тестах конфигурация переопределяется (тесты устанавливают `TESTING=True`, `SQLALCHEMY_DATABASE_URI='sqlite:///:memory:'`, `WTF_CSRF_ENABLED=False`).


## 3. АРХИТЕКТУРА И СТРУКТУРА ПРОЕКТА

- Тип архитектуры (монолит/микросервисы/MVC/и т.д.)

Проект выполнен как монолитное серверное веб-приложение на Flask. Логика приложения, модели данных и маршруты сосредоточены в одном модуле `app.py`, а представления разделены на Jinja2-шаблоны в папке `templates`. Это типичный подход «monolithic MVC-like», где модели и контроллеры находятся вместе, а представления (view) — в шаблонах.

- Полная структура файлов и папок с объяснением назначения каждого файла

На верхнем уровне проекта находятся: `app.py` — главный модуль приложения (конфигурация, модели, маршруты, CLI); `requirements.txt` — список зависимостей; `README.md` — информация о проекте; `pytest.ini` — конфигурация pytest; `tests/` — набор тестов и `templates/` — Jinja2-шаблоны для страниц. В `tests/` есть `conftest.py` (фикстуры тестового приложения и данных) и `test_app.py` (набор функциональных тестов). В `templates/` находятся шаблоны интерфейса: `base.html` (общий шаблон), `home.html`, `login.html`, `register.html`, `quiz_detail.html`, `take_quiz.html`, `quiz_result.html`, `teacher_dashboard.html`, `quiz_form.html`, `edit_quiz.html`, `question_form.html`, `subjects.html`, `subject_form.html`, `quiz_stats.html`, `my_results.html` и `error.html`.

- Как организован код внутри основных файлов (логические блоки, строки)

В `app.py` код структурирован логически по разделам: конфигурация приложения и инициализация расширений; определения моделей SQLAlchemy (User, Subject, Quiz, Question, Answer, Attempt, StudentAnswer, вспомогательная таблица `student_answer_choices`); загрузчик пользователя для `Flask-Login`; вспомогательные функции/декораторы (`teacher_required`); маршруты по функциональным областям: аутентификация, маршруты студента, маршруты преподавателя, обработчики ошибок и CLI-команды. Маршруты следуют последовательности CRUD-операций и логики прохождения теста.

- Паттерны проектирования, которые применяются

Используется ORM (Data Mapper через SQLAlchemy) и паттерн декоратора для `teacher_required`. Применён MVC-подход на уровне разделения моделей (в `app.py`) и представлений (Jinja2-шаблоны). Также присутствует шаблон «фабричного» использования `@app.cli.command` для CLI. Архитектурные паттерны более сложного уровня (Blueprints, сервисы, репозитории) не применяются — всё сведено в один модуль.


## 4. БАЗА ДАННЫХ

- Тип и название СУБД

Проект нацелен на использование реляционной СУБД через SQLAlchemy. В коде по умолчанию указан PostgreSQL (строка в `app.config['SQLALCHEMY_DATABASE_URI']`), при этом в тестах используется SQLite in-memory. Таким образом поддерживаются и Postgres, и SQLite.

- Количество таблиц

В модели определено 7 основных таблиц: `users`, `subjects`, `quizzes`, `questions`, `answers`, `attempts`, `student_answers` и дополнительная таблица связи `student_answer_choices` (association table) для M:N между `student_answers` и `answers`.

- Для КАЖДОЙ таблицы:

1) Таблица `users`
- Название: `users`.
- Поля: `id` (Integer, PK), `username` (String(80), unique, nullable=False), `email` (String(120), unique, nullable=False), `first_name` (String(50), nullable=False, default=''), `last_name` (String(50), nullable=False, default=''), `password_hash` (String(256), nullable=False), `is_teacher` (Boolean, default=False), `created_at` (DateTime, default=datetime.now(timezone.utc)).
- Назначение: хранит данные аккаунтов пользователей (студентов и преподавателей), их роль (`is_teacher`) и метаданные. Используется `UserMixin` от Flask-Login для интеграции с сессиями.
- Связи: 1:N с `Attempt` (через `attempts`), 1:N с `Quiz` (через `quizzes`) — один пользователь (преподаватель) может быть автором множества тестов; один студент может иметь много попыток.

2) Таблица `subjects`
- Название: `subjects`.
- Поля: `id` (Integer, PK), `name` (String(200), nullable=False), `description` (Text, default='').
- Назначение: каталог предметов/категорий, к которым привязываются тесты.
- Связи: 1:N с `Quiz` (через `quizzes`) — один предмет может иметь много тестов.

3) Таблица `quizzes`
- Название: `quizzes`.
- Поля: `id` (Integer, PK), `title` (String(255), nullable=False), `description` (Text, default=''), `time_limit` (Integer, default=0), `passing_score` (Integer, default=60), `is_active` (Boolean, default=True), `created_at` (DateTime, default=datetime.now(timezone.utc)), `subject_id` (Integer, FK -> subjects.id, nullable=False), `author_id` (Integer, FK -> users.id, nullable=False).
- Назначение: хранит метаданные теста (название, время, проходной балл, видимость) и связывает тест с предметом и автором.
- Связи: 1:N с `Question` (через `questions`), 1:N с `Attempt` (через `attempts`), N:1 к `Subject` и `User`.

4) Таблица `questions`
- Название: `questions`.
- Поля: `id` (Integer, PK), `text` (Text, nullable=False), `question_type` (String(10), default='single'), `order` (Integer, default=0), `explanation` (Text, default=''), `quiz_id` (Integer, FK -> quizzes.id, nullable=False).
- Назначение: хранит текст вопроса, тип (single/multiple), порядок отображения и объяснение, которое показывается после теста.
- Связи: 1:N с `Answer` (через `answers`), N:1 с `Quiz`.

5) Таблица `answers`
- Название: `answers`.
- Поля: `id` (Integer, PK), `text` (Text, nullable=False), `is_correct` (Boolean, default=False), `order` (Integer, default=0), `question_id` (Integer, FK -> questions.id, nullable=False).
- Назначение: варианты ответов для вопросов, флаг `is_correct` указывает правильные варианты.
- Связи: N:1 с `Question`; M:N с `StudentAnswer` через таблицу `student_answer_choices`.

6) Таблица `attempts`
- Название: `attempts`.
- Поля: `id` (Integer, PK), `score` (Float, default=0), `correct_count` (Integer, default=0), `total_questions` (Integer, default=0), `status` (String(15), default='in_progress'), `started_at` (DateTime, default=datetime.now(timezone.utc)), `finished_at` (DateTime, nullable=True), `student_id` (Integer, FK -> users.id, nullable=False), `quiz_id` (Integer, FK -> quizzes.id, nullable=False).
- Назначение: фиксирует каждую попытку прохождения теста студентом, результат и метрики прохождения.
- Связи: 1:N с `StudentAnswer` (через `student_answers`), N:1 с `User` и `Quiz`.

7) Таблица `student_answers`
- Название: `student_answers`.
- Поля: `id` (Integer, PK), `is_correct` (Boolean, default=False), `attempt_id` (Integer, FK -> attempts.id, nullable=False), `question_id` (Integer, FK -> questions.id, nullable=False).
- Назначение: хранит ответы студента по каждому вопросу (объект, в который привязаны выбранные варианты через M:N таблицу). Содержит признак `is_correct` для каждого ответа.
- Связи: M:N с `Answer` через `student_answer_choices` (association table); N:1 с `Attempt` и `Question`.

8) Таблица `student_answer_choices` (association)
- Название: `student_answer_choices`.
- Поля: `student_answer_id` (FK -> student_answers.id, PK составной), `answer_id` (FK -> answers.id, PK составной).
- Назначение: обеспечивает связь «многие-ко-многим» между объектом ответа студента и выбранными вариантами `Answer`.

- Общая схема связей между всеми таблицами

Схема: `User (1) — (N) Quiz` (авторство), `User (1) — (N) Attempt` (студент → попытки), `Subject (1) — (N) Quiz`, `Quiz (1) — (N) Question`, `Question (1) — (N) Answer`, `Attempt (1) — (N) StudentAnswer`, и `StudentAnswer (M) — (N) Answer` через `student_answer_choices`. Все связи реализованы через внешние ключи и ORM-relationship с каскадным удалением там, где это объявлено.


## 5. ФУНКЦИОНАЛЬНОСТЬ — РОЛЬ «ПРЕПОДАВАТЕЛЬ»
Ниже перечислены основные функции для роли преподавателя и их детальное описание на основе кода в `app.py`.

- Просмотр дашборда
- Что делает: отображает список созданных преподавателем тестов, предметы, недавние попытки студентов и базовую статистику (количество тестов, студентов, попыток, предметов).
- Маршрут: `GET /teacher`.
- Входные данные: нет (только аутентифицированный пользователь с `is_teacher=True`).
- Бизнес-логика: выбирает `Quiz` по `author_id == current_user.id`, `Subject.query.all()` и последние `Attempt`, где `Quiz.author_id == current_user.id` и `Attempt.status == 'completed'`. Считает общее число студентов (те, у кого `is_teacher=False`).
- Результат: рендер шаблона `teacher_dashboard.html` с переменными `quizzes`, `subjects`, `recent`, `total_students`.
- Проверки и валидации: декораторы `@login_required` и `@teacher_required` обеспечивают проверку авторизации/роли; дополнительных валидаций входных данных не требуется.

- Создание теста
- Что делает: отображает форму создания теста и сохраняет новый `Quiz` в базе.
- Маршрут: `GET/POST /teacher/quiz/create`.
- Входные данные (форма): `title`, `description` (необязательно), `subject_id`, `time_limit` (минуты, 0 = без лимита), `passing_score` (процент), `is_active` (checkbox).
- Бизнес-логика: при POST формирует объект `Quiz` с `author_id=current_user.id`, парсит числовые поля через `int()`, устанавливает `is_active` булево через наличие ключа в `request.form`, сохраняет в БД и перенаправляет на редактирование теста.
- Результат: создан объект `Quiz`, сообщение `flash` и редирект на `edit_quiz`.
- Проверки и валидации: минимальные; отсутствуют проверки на корректность `subject_id` (поле передаётся селектом, но отсутствует явная проверка существования subject в коде), проверяются типы данных через приведение к `int`.

- Редактирование теста
- Что делает: обновляет метаданные существующего теста (название, описание, предмет, время, проходной балл, активность).
- Маршрут: `GET/POST /teacher/quiz/<quiz_id>/edit`.
- Входные данные: те же поля, что и при создании.
- Бизнес-логика: проверяется авторство (`quiz.author_id == current_user.id`), затем значения из формы присваиваются полям объекта и `db.session.commit()` сохраняет изменения.
- Результат: обновлённый тест и flash-сообщение об успешном обновлении.
- Проверки и валидации: проверка авторства и `@teacher_required`. Валидация входных значений минимальна (подавляющийся `ValueError` не отрабатывается явным образом).

- Удаление теста
- Что делает: удаляет тест вместе с вопросами и ответами (каскадно).
- Маршрут: `POST /teacher/quiz/<quiz_id>/delete`.
- Входные данные: нет, только POST (формы с подтверждением на стороне клиента).
- Бизнес-логика: проверяется авторство, затем `db.session.delete(quiz)` и `db.session.commit()`. В моделях указаны `cascade='all, delete-orphan'`, поэтому вопросы и ответы удаляются каскадно.
- Результат: тест удалён, flash-уведомление и редирект на дашборд.
- Проверки и валидации: проверка авторства и роль преподавателя.

- Добавление вопроса
- Что делает: добавляет новый вопрос с несколькими ответами к указанному тесту.
- Маршрут: `GET/POST /teacher/quiz/<quiz_id>/question/add`.
- Входные данные: `text`, `question_type` ('single' или 'multiple'), `order`, `explanation`, и набор полей `answer_text_i` + `answer_correct_i` для вариантов.
- Бизнес-логика: создаётся `Question`, затем парсятся последовательно поля `answer_text_1`, `answer_text_2`... — для каждого непустого `answer_text_i` создаётся `Answer` с `is_correct` в зависимости от наличия `answer_correct_i`. Затем коммит.
- Результат: вопрос и ответы сохранены, уведомление и редирект на редактирование теста.
- Проверки и валидации: проверка авторства и роль преподавателя; текст вопроса обязателен на уровне шаблона (`required`), но на сервере минимальные проверки.

- Редактирование вопроса
- Что делает: обновляет текст вопроса, тип, порядок, объяснение и полностью заменяет варианты ответов.
- Маршрут: `GET/POST /teacher/question/<question_id>/edit`.
- Входные данные: те же, что и при добавлении вопроса (серия `answer_text_i` и `answer_correct_i`).
- Бизнес-логика: проверяется авторство через `q.quiz.author_id == current_user.id`; удаляются старые связи `student_answer_choices` для ответов данного вопроса, затем удаляются сами `Answer`-записи и создаются новые варианты по данным формы. Это обеспечивает консистентность связей.
- Результат: вопрос обновлён, новые ответы созданы, редирект на страницу редактирования теста.
- Проверки и валидации: проверка авторства; отсутствуют сложные проверки формата входных данных.

- Удаление вопроса
- Что делает: удаляет вопрос и связанные ответы.
- Маршрут: `POST /teacher/question/<question_id>/delete`.
- Входные данные: нет (POST), проверяется авторство.
- Бизнес-логика: `db.session.delete(q)` и `db.session.commit()` — каскад удаляет ответы, ссылки в `student_answer_choices` и т.д.
- Результат: вопрос удалён, redirect на `edit_quiz`.
- Проверки и валидации: авторство и роль преподавателя.

- Просмотр статистики теста

- Проверки и валидации: `@login_required` и логика для роли преподавателя.

- Просмотр деталей теста
- Что делает: отображает метаданные теста, параметры (время, количество вопросов, проходной балл) и кнопку «Начать тест». Также показывает предыдущие попытки пользователя.
- Маршрут: `GET /quiz/<quiz_id>`.
- Входные данные: `quiz_id` в URL.
- Бизнес-логика: проверяет, если `quiz.is_active` — доступен всем, иначе недоступен студентам (`abort(403)`), затем собирает предыдущие попытки текущего пользователя по данному тесту.

- Что делает: создаёт запись `Attempt` для текущего пользователя и редиректит на страницу прохождения теста.
- Маршрут: `POST /quiz/<quiz_id>/start`.
- Входные данные: POST без тела (любой CSRF/подтверждение на стороне клиента). Сервер использует `quiz.question_count`.
- Бизнес-логика: если у теста нет вопросов — показывает flash и не создаёт попытку; иначе создаёт `Attempt(student_id=current_user.id, quiz_id=quiz_id, total_questions=quiz.question_count)` и сохраняет.
- Результат: новая попытка в `attempts` со статусом `in_progress` и redirect на `take_quiz` (/attempt/<id>). 
- Проверки и валидации: проверка наличия вопросов, `@login_required`.

- Прохождение теста (страница вопросов)
- Что делает: отображает вопросы и варианты для конкретной попытки; если таймер есть — показывает оставшееся время; на POST вызывает завершение попытки и подсчёт результатов.
- Маршрут: `GET/POST /attempt/<attempt_id>`.
- Входные данные: при GET — attempt_id; при POST — поля формы `q_<question_id>` (для single: одно значение, для multiple: список значений). Также форма не содержит CSRF-токена (реализация простая).
- Бизнес-логика: проверяется владение попыткой `attempt.student_id == current_user.id`; если `attempt.status != 'in_progress'` — редирект на результат; вычисляется `remaining` времени на сервере (если `quiz.time_limit > 0`) путем сравнения `started_at` и текущего времени. Если POST — вызывается `_finish_attempt`.
- Результат: при успешной отправке — редирект на `quiz_result`, при истечении таймера — автоматическое завершение через серверную проверку и redirect.

- Завершение попытки и подсчёт результатов
- Что делает: сохраняет выбранные варианты в `StudentAnswer` и `student_answer_choices`, вычисляет `correct_count`, `score`, статус `completed` и `finished_at`.
- Маршрут: внутренняя функция `_finish_attempt`, вызываемая при POST `/attempt/<id>` или при тайм-ауте.
- Входные данные: из `request.form` берутся `q_<id>` поля (список или одиночный выбор).
- Бизнес-логика: для каждого вопроса: собираются `selected_ids`, вычисляется множество `correct_ids` из `q.correct_answers`. Создаётся `StudentAnswer`, привязываются выбранные `Answer` объекты (через `sa.selected_answers = selected`), определяется `is_correct` как точное совпадение множеств (`{selected} == {correct_ids}`). Если совпадают — увеличивается `correct_count`. После прохода по всем вопросам вычисляется `score` как процент: `round(correct_count / total * 100, 1)`.
- Результат: запись Attempt обновлена (`correct_count`, `score`, `status='completed'`, `finished_at`), редирект на страницу результата.
- Проверки и валидации: корректность ownership проверяется ранее; логика оценки строгая: для multiple-типов засчитывается только полный точный набор правильных ответов.

- Просмотр результатов
- Что делает: показывает итоговый экран (`quiz_result.html`) с процентами, количеством правильных ответов и детальным разбором по вопросам.
- Маршрут: `GET /attempt/<attempt_id>/result`.
- Входные данные: `attempt_id` в URL.
- Бизнес-логика: проверяется, что пользователь — владелец попытки или преподаватель (`if attempt.student_id != current_user.id and not current_user.is_teacher: abort(403)`), затем рендерятся `attempt.student_answers` и связанные сведения.
- Результат: разбор ответов, отображение объяснений к вопросам (если есть).
- Проверки и валидации: контроль доступа по владельцу/преподавателю.


## 7. АЛГОРИТМЫ И БИЗНЕС-ЛОГИКА
Подробное описание ключевых алгоритмов, реализованных в коде.

- Алгоритм аутентификации (регистрация + вход)

Регистрация (`/register`, POST): сервер собирает поля `username`, `email`, `first_name`, `last_name`, `password`, `password2`. Выполняется набор проверок: непустые поля, длина пароля >= 6, совпадение паролей, уникальность `username` и `email` (через запросы к `User.query.filter_by(...)`). При отсутствии ошибок создаётся `User`, вызывается `user.set_password(password)` (см. ниже), `db.session.add` и `commit()`, затем выполняется `login_user(user)` и редирект на главную. Ошибки передаются через `flash`.

Вход (`/login`, POST): по `username` ищется запись `User`. Если пользователь найден и `user.check_password(password)` возвращает True, вызывается `login_user(user)` и редирект на `next` параметр или на домашнюю страницу. В противном случае — `flash` с сообщением об ошибке.

- Алгоритм прохождения теста (от начала до конца)

1) Студент нажимает «Начать тест» → POST `/quiz/<id>/start` создаёт запись `Attempt` со статусом `in_progress` и `total_questions=quiz.question_count` и сохраняет в БД. 2) Редирект на `/attempt/<attempt_id>` где GET рендерит страницу с вопросами и (если задано) оставшимся временем, рассчитанным на сервере. 3) Если у теста есть `time_limit`, сервер при каждом заходе вычисляет `remaining = quiz.time_limit*60 - elapsed_seconds`, где `elapsed_seconds = now - attempt.started_at`. Если `remaining <= 0`, сервер вызывает `_finish_attempt` автоматически. 4) При ручной отправке формы (`POST /attempt/<id>`) сервер вызывает `_finish_attempt` и завершает попытку: сохраняет `StudentAnswer` объекты, заполняет связи с выбранными `Answer` и считает `correct_count` и `score`. 5) Результат сохраняется, статус меняется на `completed`, и пользователь перенаправляется на страницу результата.

- Алгоритм подсчёта баллов (с примерами для single и multiple типов)

Для каждого вопроса берутся множества: `selected_ids` — id выбранных ответов (из формы), и `correct_ids` — id правильных ответов (из модели `Question.correct_answers`). Логика начисления:
- Если множества равны (точное совпадение) → `is_correct = True` и `correct_count += 1`.
- В противном случае → `is_correct = False`.

Примеры:
- Single (1 правильный ответ): правильный ответ id=5, студент выбрал [5] → `selected_ids == correct_ids` → засчитано (1/1 → 100%). Если выбрал другой id или не выбрал — не засчитано (0%).
- Multiple (несколько правильных): правильные {2,4,7}. Студент выбрал {2,4,7} → засчитано. Если выбрал {2,4} (частичное совпадение) или {2,4,6} (есть лишний неправильный) — не засчитано. В тестах специально проверяется, что частичное совпадение даёт 0% для этого вопроса.

- Алгоритм работы таймера (клиент + сервер)

Сервер: при рендере `take_quiz` вычисляет `remaining` в секундах, если у `Quiz.time_limit > 0`, как разницу между `time_limit*60` и времени, прошедшего с `Attempt.started_at`. Если `remaining <= 0`, сервер немедленно завершает попытку через `_finish_attempt`.

Клиент: на странице `take_quiz.html` запускается JS-таймер, и каждую секунду уменьшается локальная переменная `s`. Если `s <= 0`, JS выполняет `document.getElementById('qform').submit()` — отправка формы. Также при s<=60 подсвечивается таймер как предупреждение. Серверная проверка обязательна, клиентская — вспомогательная.

- Алгоритм контроля доступа

Контроль доступа реализован с помощью `Flask-Login` и декоратора `@login_required`. Для маршрутов преподавателя применён дополнительный декоратор `@teacher_required`, который проверяет `current_user.is_authenticated and current_user.is_teacher`, в противном случае вызывает `abort(403)`. Для отдельных операций также есть проверка владения ресурсом, например `quiz.author_id != current_user.id` или `attempt.student_id != current_user.id` → `abort(403)`.

- Алгоритм хранения паролей (хеширование)

Пароли не хранятся в явном виде. При регистрации в `User.set_password` вызывается `generate_password_hash(password)` из Werkzeug и результат сохраняется в поле `password_hash`. При логине `check_password_hash` сравнивает хеш и введённый пароль. В тестах проверяется, что хеширование работает: `user.password_hash != plaintext` и `user.check_password('...')` корректно валидирует.

- Любые другие важные алгоритмы, которые есть в коде

- Каскадное удаление реализовано на уровне ORM через `cascade='all, delete-orphan'` в отношениях моделей `Quiz.questions`, `Quiz.attempts`, `User.attempts`, `User.quizzes` и т.д., что позволяет при удалении теста автоматически удалить связанные вопросы и ответы.
- При редактировании вопроса сначала очищаются записи в таблице `student_answer_choices` для ответов этого вопроса, затем удаляются старые `Answer` и создаются новые — это предотвращает висячие внешние ключи.

## 9. БЕЗОПАСНОСТЬ

- Какие угрозы предусмотрены

Рассмотренные угрозы: несанкционированный доступ к страницам (контроль ролей), хранение паролей, целостность данных при удалении (каскад), SQL-инъекции (через ORM снижается риск). Некоторые угрозы не полностью покрыты (например, CSRF и защита от брутфорса).

- Как реализована защита от каждой угрозы

- Несанкционированный доступ: `Flask-Login` + декоратор `@login_required`, собственный `@teacher_required` для проверки роли; при нарушении прав — `abort(403)` и рендер `error.html`.
- Хранение паролей: используется `generate_password_hash`/`check_password_hash` из Werkzeug, то есть пароли хранятся в виде безопасного хеша.
- SQL-инъекции: использование SQLAlchemy ORM вместо ручной конкатенации строк запросов минимизирует риск инъекций.
- Целостность данных: каскадное удаление через `cascade='all, delete-orphan'` уменьшает риск оставления «висячих» записей.

- Управление сессиями

Сессии управляются через `Flask-Login` и стандартные cookie-сессии Flask. Используется `app.config['SECRET_KEY']` (значение читается из переменных окружения через `.env`), что необходимо для подписи сессионных cookie. Нет в коде явной настройки параметров cookie (например `SESSION_COOKIE_SECURE`, `REMEMBER_COOKIE_DURATION`) — эти параметры остаются по умолчанию.

- Контроль доступа

Осуществляется через `@login_required` и собственный `@teacher_required`. Дополнительно контроль права владения ресурсом (проверки `quiz.author_id` и `attempt.student_id`) предотвращает доступ к чужим данным. Маршруты, где требуется роль преподавателя, защищены и возвращают 403 при нарушении.


## 10. ЧТО ИЗМЕНИЛОСЬ / ЧТО НОВОЕ
Сравнение с типичным минимальным Flask-приложением и выделение особенностей.

- Какие нестандартные решения применены

Всё реализовано в одном файле `app.py` (модель + контроллер), что упрощает структуру, но снижает модульность. Для хранения ответов студентов введена промежуточная сущность `StudentAnswer` и ассоциативная таблица `student_answer_choices`, что даёт гибкость при хранении множественных выбранных вариантов. При редактировании вопроса авторы сначала очищают таблицу связей вручную, что необычно, но гарантирует удаление старых связей.

- Что было улучшено или добавлено (если видно по коду)

В коде видно внимательное отношение к тестам: в `tests/conftest.py` скрипты создают фикстуры с явными `commit()` между созданием сущностей, чтобы обеспечить корректные ID и работоспособность тестов. Также реализована клиентская логика выбора ответов и таймер, что повышает UX. Наличие CLI-команд `init-db` и `create-teacher` упрощает развёртывание и администрирование.

- Сложные места в коде, которые требуют особого объяснения в пояснительной записке

1) Механика сохранения и замены ответов при редактировании вопроса: нужно пояснить, почему удаляются связи `student_answer_choices` и затем `Answer` перед созданием новых — это важно для целостности данных. 2) Логика оценки — строгая проверка множеств правильных ответов — стоит объяснить в связи с педагогической политикой (частичные баллы не начисляются). 3) Обработка таймера: серверная и клиентская части работают вместе, и в пояснительной записке важно отметить необходимость синхронизации и доверия только серверной проверке.


## 11. ТЕСТИРОВАНИЕ

- Есть ли тесты в проекте (файлы, фреймворк)

Да — в папке `tests/` присутствуют `conftest.py` и `test_app.py`. Тесты написаны с использованием `pytest` и `pytest-flask`.

- Что именно тестируется

Функциональные сценарии: регистрация, вход/выход, контроль доступа (студент не может зайти в панель преподавателя, нельзя просматривать чужие попытки и т.п.), создание тестов и вопросов преподавателем, поведение при прохождении тестов (правильные/неправильные ответы), обработка multiple-choice и частичных совпадений, каскадное удаление теста, статистика, граничные случаи (пустая форма), проверка хеширования паролей.

- Результаты (если можно определить по коду)

Тесты утверждаются ожидаемыми состояниями базы данных и ответами клиентов (`status_code`, содержание HTML). Код тестов показывает, что приложение должно проходить эти сценарии; сами результаты тестов в репозитории не представлены, но тесты тщательно покрывают основные сценарии и edge-cases.

- Типы тестирования (модульное, интеграционное и т.д.)

В основном это интеграционные / функциональные тесты, выполняемые через тестовый клиент Flask и in-memory SQLite БД (`sqlite:///:memory:`) в фикстуре `test_app`. Это близко к интеграционному тестированию, поскольку проверяется взаимодействие маршрутов, ORM и логики приложения вместе.


## 12. ВОЗМОЖНЫЕ УЛУЧШЕНИЯ
Предложения по доработке проекта (3–5 пунктов) с обоснованием.

1) Ввести CSRF-защиту: сейчас формы реализованы без CSRF-токенов (в тестах `WTF_CSRF_ENABLED=False`), это уязвимость для атак подделки запросов. Рекомендуется включить Flask-WTF/CSRFProtect и добавить токены во все формы.

2) Разделить приложение на модули/blueprints: переместить модели в `models.py`, маршруты — в отдельные `blueprints` (`auth`, `student`, `teacher`). Это улучшит масштабируемость и поддерживаемость проекта.

3) Добавить подтверждение email и сброс пароля: сейчас регистрация создаёт активный аккаунт без проверки. Для реального использования полезно добавить верификацию email и восстановление пароля.

4) Улучшить валидацию форм и обработку ошибок: использовать WTForms для валидации и обработки ошибок, добавить обработку исключений и пользовательские сообщения для некорректных входных данных.

5) Логирование и мониторинг: добавить структурированное логирование (например, через `logging`), метрики и возможность экспорта статистики (CSV/Excel) для преподавателя.
